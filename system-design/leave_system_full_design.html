Automated Leave Approval System — Full Design Reference

Request lifecycle — step by step

1
Client → NGINX. Request hits NGINX. Rate limiter checks IP bucket (100 req/min). Exceeds limit → 429. Passes → routed to least-loaded FastAPI instance via upstream pool. SSL terminated here.
2
NGINX → FastAPI. FastAPI receives POST /leave/apply. Request is stateless — any instance can handle it.
3
Auth check. JWT token validated (signature + expiry). Role extracted (student | faculty | admin). Unauthorized → 401. Role mismatch for action → 403.
4
Idempotency check. Compute key = SHA256(user_id + leave_type + start_date + end_date). Check Redis: GET idempotency:{key}. If exists → return cached response immediately (duplicate suppressed). If absent → SET idempotency:{key} "processing" EX 86400 NX (atomic, 24h TTL).
5
Persist PENDING. Insert row in leave_requests with status = PENDING. Write audit log entry: system → PENDING. Commit transaction. Now durable — even if the process crashes, request is recoverable.
6
Rule engine evaluation. Rules loaded from Redis cache (TTL 5 min, backed by DB). Engine evaluates sequentially by priority. Three outcomes:
APPROVED → update DB, write audit log, enqueue notification, return 200 {status: "APPROVED"} (<200ms)
REJECTED → update DB, write audit log, enqueue notification, return 200 {status: "REJECTED"} (<200ms)
UNCERTAIN → push job to Redis queue, update idempotency key to request_id, return 202 {status: "PROCESSING", request_id}
7
AI worker (async). Worker pops job from Redis queue. Builds context prompt. Calls AI model API. Receives APPROVED/REJECTED decision with reasoning. Updates DB. Writes audit log. Enqueues notification. Updates idempotency key with final result.
8
Notification service. Separate worker consumes notification queue. Sends email/push asynchronously. Retries on failure. Never blocks the main request path.
9
Frontend polling. Client polls GET /leave/status/{request_id} every 3s (max 5 attempts). Response served from Redis cache first. If cache miss, reads DB and repopulates cache. Admin can override at any point via PATCH /leave/{id}/override.

Schema design

-- Users
CREATE TABLE users (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name        TEXT NOT NULL,
  email       TEXT UNIQUE NOT NULL,
  role        TEXT CHECK (role IN ('student','faculty','admin')),
  department  TEXT,
  created_at  TIMESTAMPTZ DEFAULT now()
);

-- Leave requests
CREATE TABLE leave_requests (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID NOT NULL REFERENCES users(id),
  leave_type      TEXT NOT NULL,  -- 'medical', 'casual', 'exam', etc.
  start_date      DATE NOT NULL,
  end_date        DATE NOT NULL,
  days_requested  INT GENERATED ALWAYS AS (end_date - start_date + 1) STORED,
  reason          TEXT,
  status          TEXT DEFAULT 'PENDING'
                  CHECK (status IN ('PENDING','APPROVED','REJECTED','PROCESSING')),
  decision_source TEXT CHECK (decision_source IN ('rule_engine','ai','admin')),
  ai_reasoning    TEXT,
  idempotency_key TEXT UNIQUE,
  created_at      TIMESTAMPTZ DEFAULT now(),
  updated_at      TIMESTAMPTZ DEFAULT now()
);

-- Audit log (append-only)
CREATE TABLE audit_logs (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  entity_type  TEXT NOT NULL,         -- 'leave_request'
  entity_id    UUID NOT NULL,
  actor_id     UUID REFERENCES users(id),
  actor_type   TEXT,                  -- 'system', 'rule_engine', 'ai', 'admin'
  action       TEXT NOT NULL,         -- 'status_change', 'override'
  prev_state   JSONB,
  new_state    JSONB,
  metadata     JSONB,                 -- rule matched, AI confidence, etc.
  created_at   TIMESTAMPTZ DEFAULT now()
);

-- Configurable rules
CREATE TABLE leave_rules (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  rule_key       TEXT UNIQUE NOT NULL, -- 'max_auto_approve_days'
  rule_value     JSONB NOT NULL,       -- {"value": 3, "roles": ["student"]}
  description    TEXT,
  priority       INT DEFAULT 100,
  is_active      BOOLEAN DEFAULT true,
  updated_by     UUID REFERENCES users(id),
  updated_at     TIMESTAMPTZ DEFAULT now()
);

-- Leave balance (per user per year)
CREATE TABLE leave_balances (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES users(id),
  leave_type  TEXT NOT NULL,
  year        INT NOT NULL,
  total       INT NOT NULL,
  used        INT DEFAULT 0,
  UNIQUE(user_id, leave_type, year)
);

SELECT * FROM leave_requests lr
JOIN users u ON u.id = lr.user_id

Use keyset (cursor) pagination — OFFSET degrades at scale because the DB still scans skipped rows.

Redis key namespaces

Three completely separate concerns — cache, queue, idempotency — each with distinct key patterns and TTLs.

1. Cache keys

leave:status:{request_id}     → JSON {status, updated_at, ...}  TTL: 60s
leave:balance:{user_id}:{year} → JSON {used, total}               TTL: 300s
rules:active                  → JSON array of rule objects         TTL: 300s
user:profile:{user_id}        → JSON {role, department}           TTL: 600s

2. Queue keys (Redis List)

queue:ai_jobs               → LPUSH to enqueue, BRPOP to consume (blocking pop)
queue:notifications          → LPUSH to enqueue, BRPOP to consume
queue:dlq:ai_jobs            → Dead letter queue — jobs exceeding max retries
queue:dlq:notifications      → DLQ for failed notifications

Job payload structure:

{
  "job_id":     "uuid",
  "request_id": "uuid",
  "user_id":    "uuid",
  "leave_type": "medical",
  "start_date": "2025-06-01",
  "end_date":   "2025-06-05",
  "reason":     "...",
  "attempt":    1,
  "enqueued_at": "ISO8601"
}

3. Idempotency keys

idempotency:{sha256_key}    → "processing" | "{request_id}"    TTL: 86400s (24h)

Set with NX flag (only if not exists) — atomic, no race condition. First request wins. Duplicate requests within 24h get the original response.

4. Rate limiting keys (NGINX / application level)

ratelimit:ip:{ip_address}    → counter                           TTL: 60s
ratelimit:user:{user_id}     → counter                           TTL: 3600s

AI worker design

Worker loop (pseudocode)

while True:
    job = redis.BRPOP("queue:ai_jobs", timeout=5)
    if not job:
        continue

    data = parse(job)

    if data["attempt"] > MAX_RETRIES:  # MAX_RETRIES = 3
        redis.LPUSH("queue:dlq:ai_jobs", job)
        log_error("Job exceeded max retries", data["job_id"])
        continue

    try:
        decision = call_ai_model(data)   # APPROVED | REJECTED + reasoning
        update_db(data["request_id"], decision)
        write_audit_log(data, decision, actor="ai")
        update_cache(data["request_id"], decision)
        update_idempotency(data["idempotency_key"], data["request_id"])
        enqueue_notification(data, decision)

    except AIServiceException as e:
        delay = 2 ** data["attempt"]  # 2s, 4s, 8s
        data["attempt"] += 1
        data["retry_after"] = now() + delay
        schedule_retry(data, delay)   # re-push after delay
        log_warning("AI call failed, retry scheduled", delay=delay)

    except Exception as e:
        redis.LPUSH("queue:dlq:ai_jobs", job)
        log_error("Unrecoverable error", e)

DLQ processing

DLQ jobs are not lost — they sit in queue:dlq:ai_jobs. An ops engineer can inspect them via admin API, fix the root cause, and replay them: BRPOP dlq → LPUSH queue:ai_jobs. Alert fires when DLQ size exceeds threshold (see Metrics tab).

Fallback if AI is unavailable

If AI service is down after all retries, the request stays PROCESSING and a manual review flag is set. Admin is notified. SLA: admin must review within 24h. This is preferable to auto-rejecting a legitimate request.

Configurable rule engine

Rules are stored in the leave_rules DB table and cached in Redis for 5 minutes. No hardcoding. Changes take effect within one cache TTL without a deploy.

Rule evaluation flow

def evaluate(request, user, balance, rules) -> "APPROVED" | "REJECTED" | "UNCERTAIN":
    # Sort rules by priority (lower number = higher priority)
    for rule in sorted(rules, key=lambda r: r.priority):
        if not rule.applies_to(user.role, request.leave_type):
            continue

        result = rule.evaluate(request, user, balance)
        if result in ("APPROVED", "REJECTED"):
            return result   # First decisive rule wins

    return "UNCERTAIN"      # No rule was decisive → send to AI

UNCERTAIN triggers (sent to AI)

Read strategy: cache-aside (lazy loading)

def get_leave_status(request_id):
    cached = redis.GET(f"leave:status:{request_id}")
    if cached:
        return parse(cached)          # cache hit

    record = db.query(request_id)    # cache miss → read DB
    redis.SETEX(f"leave:status:{request_id}", 60, serialize(record))
    return record

Cache invalidation

Invalidation happens immediately on any state change — never wait for TTL expiry when accuracy matters:

def update_leave_status(request_id, new_status):
    db.update(request_id, status=new_status)         # 1. write DB first
    redis.DELETE(f"leave:status:{request_id}")       # 2. invalidate cache
    redis.SETEX(f"leave:status:{request_id}", 60,    # 3. repopulate
                serialize(new_status))

Write-through (not write-back) — always write DB first. If Redis write fails, the DB is the source of truth. Cache staleness is bounded by TTL in the worst case.

Cache warming on startup

On worker/API startup, preload rules:active into Redis. Avoids cold-start latency spike when a new instance spins up during a traffic burst.

Scaling behavior

1K users — single-node baseline

Single FastAPI instance, single PostgreSQL, single Redis. Rule engine handles ~95% of requests in <50ms. No queue depth issues. One AI worker sufficient. Main concern: database connection pool sizing (set pool size = 20–30).

Bottleneck: None at this scale. Keep it simple.

10K users — horizontal API scaling

Add 3–5 FastAPI instances behind NGINX (round-robin). Stateless design means no session affinity needed. Redis becomes critical — all instances share the same cache, idempotency store, and queues. Add PostgreSQL read replica for admin list queries (write to primary, read listings from replica). Add 2–3 AI workers. NGINX handles ~5K req/s easily.

Bottleneck: PostgreSQL write throughput (especially audit logs). Mitigate with connection pooling (PgBouncer) and batching audit writes.

1M users — serious distributed system

At this scale, the architecture changes materially:

  • PostgreSQL: Partition leave_requests by created_at (monthly). Partition audit_logs separately. Add read replicas per region. Consider archiving old records to cold storage.
  • Redis: Redis Cluster with 3+ shards. Separate Redis instances for queue vs cache vs idempotency (different eviction policies).
  • Workers: Scale AI workers independently from API (they have different resource profiles — API is CPU-light, workers are I/O-heavy waiting on AI API).
  • API instances: 20–50 FastAPI processes. NGINX upstream pool auto-managed.
  • Queue: Redis lists still work at this scale, but consider migrating to a purpose-built queue (Celery with Redis backend, or eventually Kafka). This is when the "no Kafka" constraint starts to hurt.
  • CDN: Static assets and public-facing status pages behind CDN.
  • Rate limiting: Move from NGINX local state to Redis-backed rate limiting (distributed, consistent across instances).

Bottleneck: AI API rate limits (external dependency). Implement request coalescing, priority queuing (faculty requests before student?), and circuit breaker pattern on AI calls.

Design tradeoffs — interview-ready explanations

Why Redis (not Kafka)?

Chosen: Redis lists (LPUSH/BRPOP) for queuing.

Why: The constraint said no Kafka, but the reasoning is genuine: at <10K users, Kafka adds operational overhead (ZooKeeper/KRaft, broker management, consumer group coordination) that isn't justified. Redis already exists in the stack for caching and idempotency. Using it for queuing avoids adding a new infrastructure component. Redis lists give FIFO semantics, blocking pop, and sufficient throughput for this scale.

Tradeoff: Redis lists lack Kafka's replay capability, offset management, and fan-out to multiple consumer groups. If we needed to replay AI decisions or have multiple independent consumers of the same event, we'd pay for Kafka. At current scale, YAGNI — we don't need it.

Why async processing for AI?

Why: AI inference takes 2–15 seconds. Blocking the HTTP request thread for that duration would exhaust the FastAPI worker pool under any meaningful load. By returning 202 PROCESSING immediately, we decouple the user-facing response latency from the AI processing latency. The user gets a fast response, the system stays available, and the AI decision arrives asynchronously.

Tradeoff: Polling complexity on the frontend. We mitigate with WebSockets as a future option — but polling every 3s is simple and sufficient for a leave approval system where decisions aren't time-critical to the second.

Why stateless backend?

Why: Stateless FastAPI instances share no in-process state — no session data, no in-memory caches specific to one instance. All shared state lives in Redis and PostgreSQL. This means any instance can handle any request, NGINX can route freely, and we can add/remove instances without sticky sessions or coordination. Horizontal scaling becomes trivially addable.

Tradeoff: Every request pays a Redis round-trip for auth token validation and cache reads. This is ~0.5–2ms and well within budget. The scalability benefit far outweighs this cost.

DB vs cache tradeoffs

DB is the source of truth, always. We write to DB before updating cache. If Redis fails, the system degrades gracefully — every request falls through to DB. Slightly higher latency, but no data loss and no stale decisions.

Cache serves reads. Status polling (the most frequent read) hits Redis first. Cache miss ratio should be <30%. If it's higher, TTLs are too short or cache is being evicted due to memory pressure.

Never cache mutable data without invalidation. Leave status changes frequently during processing. We delete-then-repopulate on every state change rather than relying on TTL expiry — a 60s stale status is acceptable for the frontend, but we don't want an admin to see wrong data after their own override action.

Rule engine vs AI: why two layers?

The rule engine handles the 80% of cases that are deterministic: trivially short leaves, clearly invalid requests, balance exhaustion. These get decisions in <50ms at near-zero cost. The AI handles the nuanced 20%: complex reasons, edge cases, conflicting signals. Sending all requests to AI would add 2–15s latency and cost to every request unnecessarily. This hybrid approach keeps p95 latency low while preserving flexibility for complex cases.
